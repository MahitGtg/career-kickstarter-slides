Part 02 · Auth

Client Authentication

Every connection must login before any other message

01 Look up users[username] from data.json
read-only · loaded at startup
02 Compare password
03 Pass → sessions[handler] = username
Fail → send error + close connection
04 Every subsequent message checks sessions[handler]
05 On disconnect → del sessions[handler]

users dict

dict[username → { password, full_name }]

Loaded once at startup · never mutated

sessions dict

dict[handler → username]

Written on login · deleted on disconnect

Part 02 · Insert Order

Step 2 — 6 Risk Checks

All must pass — any fail → reject immediately

# Condition Scope
1 user_outstanding_qty + qty ≤ max User global
2 len(msg_timestamps) < rate_limit User rate
3 instr_outstanding_qty + qty ≤ max User × Instr
4 instr_outstanding_amt + notional ≤ max User × Instr
5 sum(qty_window) + qty ≤ rolling_limit Rolling
6 sum(amt_window) + amt ≤ rolling_limit Rolling
Any check fails → send error response · no risk state changed · no message to OB

Part 02 · Core Design Decision

Why Pessimistic Update?

The concurrency problem on a single-threaded gateway

Without pessimistic update

1. Client sends Order A (qty=900)

2. Risk check: 0 + 900 ≤ 1000 ✓

3. Forwarded to OB — risk NOT updated

4. Client sends Order B (qty=900)

5. Risk check: 0 + 900 ≤ 1000 ✓ — WRONG

6. Both reach OB · 1800 resting · limit blown

With pessimistic update

1. Client sends Order A (qty=900)

2. Risk check: 0 + 900 ≤ 1000 ✓

3. Risk updated: outstanding = 900

4. Forwarded to OB

5. Client sends Order B (qty=900)

6. 900 + 900 > 1000 — REJECTED

Risk engine is always ahead of OB — no order gets processed that can't be processed

Part 02 · Insert Order

Step 3 — Pessimistic Update

Update risk state before forwarding — don't wait for OB confirmation

user_outstanding_qty[username]                     += quantity
instrument_outstanding_qty[(username, symbol)]     += quantity
instrument_outstanding_amount[(username, symbol)]  += quantity * price * multiplier

user_msg_timestamps[username].append(now)
instrument_order_qty_window[(username, symbol)].append((now, quantity))
instrument_order_amount_window[(username, symbol)].append((now, amount))

Why not update on cancel?

Cancel might fail — order may already be filled

Deductions happen downstream on confirmed cancel only

The guarantee

Any concurrent order sees the already-incremented qty

Limits can never be exceeded regardless of timing

Part 02 · Insert Order

Step 4 — Dispatch to OB

Assign unique internal ID · store correlation · forward

ob_request_id = next(_ob_id_counter)

pending_requests[ob_request_id] = {
    "handler":           handler,
    "client_request_id": request_id,
    "username":          username,
    "symbol":            symbol,
    "price":             price,
    "quantity":          quantity,
}

Sends InsertOrderRequest { request_id: ob_request_id, order_book_id, side, price, qty } to OB over TCP

When OB replies, gateway looks up pending_requests[ob_request_id] to find the original client handler and route the response back

Part 02 · Cancel Order

Cancel Order Flow

Rate check only — no outstanding qty update

01 Resolve symbol → get order_book_id
02 Message rate check only
Cancels count as messages per protocol spec
03 No other risk DS change
Cancel might fail — no deduction until OB confirms
04 Forward to OB with new ob_request_id

Why no pessimistic deduction?

Insert adds pessimistically — worst case the order rests

Cancel is the opposite — we can't assume it succeeds

If we deducted and the cancel failed, the risk engine becomes optimistic — could allow over-limit orders
