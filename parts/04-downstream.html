Part 04 · Downstream

Case 1 — Insert Errored

OB rejected the order — revert pessimistic update

We updated risk before forwarding, but OB said no. Undo everything.

user_outstanding_qty[username]                     -= quantity
instrument_outstanding_qty[(username, symbol)]     -= quantity
instrument_outstanding_amount[(username, symbol)]  -= quantity * price * multiplier

Also remove entries from rolling windows: qty_window, amt_window, msg_timestamps

Part 04 · Downstream

Case 2 — Cancel Succeeded

OB confirms cancel — now safe to deduct

Look up open_orders[order_id] for the original price and quantity.

info = open_orders[order_id]

user_outstanding_qty[username]                     -= info.quantity
instrument_outstanding_qty[(username, symbol)]     -= info.quantity
instrument_outstanding_amount[(username, symbol)]  -= info.quantity * info.price * multiplier

del open_orders[order_id]

This is why we didn't deduct upstream — we waited for OB confirmation

Part 04 · Downstream

Case 3 — Order Traded

Matched qty no longer rests — deduct from outstanding

user_outstanding_qty[username]                     -= trade.quantity
instrument_outstanding_qty[(username, symbol)]     -= trade.quantity
instrument_outstanding_amount[(username, symbol)]  -= trade.quantity * trade.price * multiplier

Filled qty was already counted from the pessimistic add — now that it traded, we release capacity

Part 04 · Performance Trade-offs

PriceDepthBook — Current Design

Data structure: dict[float, int] — unsorted hash map per side

on_order_inserted() O(1)
on_trade() O(1)
on_order_cancelled() O(1)
get_bids() O(n log n) ← sorted() on every call
get_asks() O(n log n) ← sorted() on every call
get_order() O(1)

Reads are the bottleneck — get_bids / get_asks are called on every order book change to push market data to all subscribers

Part 04 · Optimization Path

SortedDict — Sorted at Insert Time

Replace dict with SortedDict from sortedcontainers — B-tree backed, always sorted

on_order_inserted() O(log n) was O(1)
on_trade() O(log n) was O(1)
on_order_cancelled() O(log n) was O(1)
get_bids() O(n) ← already sorted, just iterate
get_asks() O(n) ← already sorted, just iterate
get_order() O(1) unchanged

Trade-off: pay O(log n) on writes to get O(n) reads — worth it when reads vastly outnumber writes (market data pushed to every subscriber on every change)

Part 04 · Market Data Fairness

Current Approach — Naive Iteration

L1 / L2 updates broadcast to subscribers in a fixed for-loop

for client in subscribers:
    client.send(market_data)      # always same order
1st CLIENT A always receives first
2nd CLIENT B slight delay
3rd CLIENT C more delay
4th CLIENT D always receives last

Unfair — Client A has a systematic latency advantage on every single update. In trading, nanoseconds matter.

Part 04 · Market Data Fairness

Fairer Distribution Strategies

Two approaches to eliminate systematic advantage

① Round-Robin Rotation

start = cycle_counter % len(subscribers)
rotated = subscribers[start:] + subscribers[:start]
for client in rotated:
    client.send(market_data)
cycle_counter += 1
Cycle 1 A B C D
Cycle 2 B C D A
Cycle 3 C D A B
Cycle 4 D A B C

② Random Shuffle

random.shuffle(subscribers)
for client in subscribers:
    client.send(market_data)

Round-robin is deterministic and guarantees equal treatment over N cycles. Random shuffle provides fairness in expectation but not per-cycle.
